# 中文

## 1. 平台概览

AI-Ops 是一个多智能体 K8s 运维平台，通过 7 个 AI Agent 协作实现告警感知、故障诊断、自动修复、效果验证的完整闭环。

| 页面 | 路由 | 功能 |

## 2. 日常运维操作

### 2.1 故障诊断

1. 进入**对话台**
2. 选择目标机器（机器选择器下拉）
3. 描述问题："payment-service 的 Pod 一直 CrashLoopBackOff，帮我排查"
4. Agent 自动执行：
   - Observe → 采集 Pod 状态/日志/事件
   - Diagnose → 分析根因（假设 + 置信度）
   - Knowledge → 检索相似历史案例
5. Agent 给出诊断结果和建议

### 2.2 执行修复操作

1. Agent 诊断完成后，会输出修复计划（蓝色卡片）
2. L0-L1 操作自动执行
3. L2+ 高风险操作弹出红色审批卡片 → 点击**批准**或**拒绝**
4. 修复后自动验证（检查 Pod/服务状态）
5. 验证失败自动调整策略重试（最多 3 轮）

### 2.3 直接执行命令

在输入框直接输入命令即可，系统会自动识别为"变更操作"并直接执行：

```
bash /root/hello.sh           # Linux 脚本执行
kubectl get pods -n default   # K8s 查询
systemctl restart docker      # 服务管理
```

- Linux 命令：执行完直接返回结果，不触发 K8s 集群检查
- K8s 命令：执行后做轻量验证（只查受影响的资源）

### 2.4 查看仪表盘

1. 进入 `/dashboard`
2. 30 秒自动刷新
3. 查看：会话总数/成功率、意图分布图、各 Agent 消息数、集群节点/Pod 状态

### 2.5 审计与回溯

1. 进入 `/audit`
2. 可按意图/状态筛选会话
3. 点击**查看**进入详情：
   - **消息标签**：按时间线展示完整消息，支持播放/暂停/调速回放
   - **DAG 标签**：查看 Agent 决策路径流程图
4. 点击**导出 CSV**下载审计报表

### 2.6 巡检管理

1. 进入 `/inspection`
2. 5 个默认巡检任务：
   - Pod 健康检查 (30min)
   - 节点资源检查 (15min)
   - 磁盘空间检查 (60min)
   - 内存使用检查 (30min)
   - 服务状态检查 (60min)
3. 开关按钮启用/禁用任务
4. 点击**立即执行**手动触发
5. 告警匹配到模式自动推送到钉钉

### 2.7 知识库管理

1. 进入 `/knowledge`
2. **语义搜索**：输入问题，向量检索匹配案例
3. **手动上传**：填写标题/内容/标签
4. **自动生成**：输入已完成会话 ID，LLM 提取症状→根因→修复链生成结构化案例

### 2.8 多机器运维

1. 进入 `/machines` 添加新机器
2. 点击**检测连接**验证 SSH 可达性
3. 设为默认后，对话台**机器选择器**自动选中
4. 在对话台切换机器后，所有命令自动路由到选中的目标

### 2.9 用户管理

1. 进入 `/users`（需 admin+ 权限）
2. 新建用户：用户名 + 密码 + 角色
3. 编辑用户：角色/状态/邮箱/密码
4. 角色说明：
   - `viewer`：只读集群查询
   - `operator`：查看 + L1 操作需审批
   - `admin`：可执行 L1-L2 操作，管理用户
   - `superadmin`：全部权限

## 3. 快捷键

| 快捷键 | 功能 |

## 4. Agent 流水线说明

| Agent | 职责 | 工具 |
|-------|------|------|
| Orchestrator | 意图分类 + 路由分发 + 结果汇总 | LLM |
| Observe | 数据采集 + 异常检测 | kubectl_exec, linux_exec, prometheus_query |
| Diagnose | 假设生成 + 根因分析 | LLM + RAG |
| Knowledge | 知识库检索 (RAG) | ChromaDB 向量搜索 |
| Remedy | 修复计划 + 命令执行 + HITL 审批 | kubectl_exec, linux_exec, risk_classifier |
| Verify | 修复验证 + 重试决策 | kubectl_exec, linux_exec |
| Consolidation | 会话结束后自动生成案例入库 | LLM + MySQL + ChromaDB |

## 5. 风险等级说明

| 等级 | 描述 | 示例操作 | 审批规则 |
|------|------|----------|----------|
| L0 | 只读 | kubectl get/describe/logs, ls/df/ps/free | 所有角色自动 |
| L1 | 安全写 | touch/mkdir/echo/printf, kubectl apply/create | viewer/operator 需审批 |
| L2 | 破坏性 | kubectl delete/scale, systemctl restart, kill | admin+ 自动 |
| L3 | 危险 | kubectl exec/drain, iptables, docker exec | superadmin 自动 |
| L4 | 灾难 | kubectl delete ns, reboot, dd, fdisk | 始终需审批 |

## 6. 通知配置

1. 进入 API 文档 `/docs`
2. `PUT /api/notify/config` 配置钉钉 webhook
3. `POST /api/notify/test` 发送测试消息
4. 可配置通知类型：异常/修复/验证失败/巡检告警

## 2. Daily Operations

### 2.1 Fault Diagnosis

1. Go to **Chat**
2. Select target machine from dropdown
3. Describe the issue: "payment-service pods are CrashLoopBackOff, help me investigate"
4. Agents automatically:
   - Observe → collect pod status/logs/events
   - Diagnose → root cause analysis
   - Knowledge → retrieve similar cases
5. View diagnostic results and recommendations

### 2.2 Executing Repairs

1. After diagnosis, repair plan appears (blue card)
2. L0-L1 operations auto-execute
3. L2+ operations show red approval card → click **Approve** or **Reject**
4. Auto-verify after repair
5. Failed verification → auto-retry with adjusted strategy (up to 3 rounds)

### 2.3 Direct Command Execution

Type commands directly; the system auto-routes:
```
bash /root/hello.sh           # Linux script
kubectl get pods -n default   # K8s query
systemctl restart docker      # Service management
```
- Linux commands: execute → return result (no K8s checks)
- K8s commands: execute → light verify (affected resources only)

### 2.4 Dashboard

1. Go to `/dashboard` (auto-refreshes every 30s)
2. Monitor: session stats, success rate, intent distribution, agent activity, cluster health

### 2.5 Audit & Replay

1. Go to `/audit`
2. Filter by intent/status
3. Click **View** for details:
   - **Messages tab**: full message timeline with play/pause/speed controls
   - **DAG tab**: agent decision path visualization
4. **Export CSV** for compliance reports

### 2.6 Inspection Management

1. Go to `/inspection`
2. 5 default tasks with configurable intervals and alert patterns
3. Toggle tasks on/off
4. Click **Run Now** for manual trigger
5. Alerts match patterns → DingTalk notification

### 2.7 Multi-Machine Operations

1. Go to `/machines` to add SSH targets
2. Click **Check** to verify connectivity
3. Set default machine for quick selection in chat
4. Machine selector in chat input area controls routing

### 2.8 User Management

1. Go to `/users` (admin+ only)
2. Create/edit/delete users
3. Four roles: viewer, operator, admin, superadmin

## 4. Agent Pipeline

| Agent | Role | Tools |
|-------|------|-------|
| Orchestrator | Intent classification + routing + summary | LLM |
| Observe | Data collection + anomaly detection | kubectl_exec, linux_exec, prometheus_query |
| Diagnose | Hypothesis + root cause analysis | LLM + RAG |
| Knowledge | RAG retrieval | ChromaDB vector search |
| Remedy | Repair plan + execution + HITL approval | kubectl_exec, linux_exec, risk_classifier |
| Verify | Post-repair verification + retry decisions | kubectl_exec, linux_exec |
| Consolidation | Auto-generate cases from sessions | LLM + MySQL + ChromaDB |

## 5. Risk Levels

| Level | Description | Examples | Approval |
|-------|-------------|----------|----------|
| L0 | Read-only | kubectl get/describe/logs, ls/df/ps | Auto (all roles) |
| L1 | Safe write | touch/mkdir/echo, kubectl apply/create | viewer/operator need approval |
| L2 | Destructive | kubectl delete/scale, systemctl restart | Auto (admin+) |
| L3 | Dangerous | kubectl exec/drain, iptables | Auto (superadmin) |
| L4 | Catastrophic | kubectl delete ns, reboot, dd | Always needs approval |
